outpost

Hand off your Claude Code session to a remote machine so it keeps working headlessly while you’re away. Bring the result back when you’re ready.

View on GitHub Try it

I’m pairing with Claude on my laptop. I need to step away — close the lid, go to a meeting, get on a flight. I want the agent to keep working autonomously on my configured dev box, picking up exactly where we left off.

When I’m back, I type /bring-back and resume work locally with the remote’s progress applied. The agent on the remote is fully headless — there’s nothing to attach to, by design. “Work while I’m away” is the whole point.

How it works

1

Type /send-away

Right inside Claude. Outpost ships your project to your dev box and launches a headless agent there with your full session context.

2

Walk away

Close your laptop, get on the train. The agent keeps working — thinking, editing, running tools — until it decides the task is done.

3

Type /bring-back

Stops the remote agent, pulls back its commits, uncommitted edits, and updated session transcript. You resume locally where it left off.

What it looks like

claude — ~/projects/my-app
> /send-away ✓ Project: -home-alice-projects-my-app ✓ Worktree reconciled to feature/x @ a3f9c1 ✓ Uncommitted: 12 hunks across 4 files / 3 untracked files ✓ 2 agents launched headlessly: - 4f8a2e1c → PID 31204 - 9c7d1f3b → PID 31207 ✓ Run `outpost bring-back` when you’re back. # … hours later, lid closed, agent kept going … > /bring-back ✓ Project: -home-alice-projects-my-app ✓ 14 commits fast-forwarded ✓ 23 hunks, 6 untracked files applied ✓ 2 session files pulled

What you get

Fully transparent

Outpost mounts a FUSE filesystem at ~/.claude/ — every read and write passes straight through to a real directory on disk at kernel speed. Never in your way, incredibly efficient.

Brings work back, not just code

Bring-back pulls everything: new commits via git, uncommitted edits applied as a patch, untracked files via tar, and the new session transcript. Your local repo ends up looking like you did the work yourself.

Multiple Claude sessions, parallel

If you have several Claude windows open on the same project, each one becomes its own headless agent on the remote. They run in parallel, each with their own log file, each independently brought back.

Try it

Install

Clone the repo and run the install script. It builds outpost, drops a default config at ~/.config/outpost/config.ini, and starts it as a user service so it’s up every time you log in.

git clone https://github.com/ralt/outpost
cd outpost && ./scripts/install.sh

Point it at your dev box

Edit the [remote] section of ~/.config/outpost/config.ini:

# ~/.config/outpost/config.ini
[remote]
host = alice@dev-box.example.com
port = 22
identity_file = /home/alice/.ssh/id_ed25519

Full setup details — SSH access, matching $HOME, what binaries need to be on PATH, and a troubleshooting table mapped to every error code outpost can return — live in the remote setup guide.

Send-away

From any Claude project, type /send-away. Outpost ships the project to your dev box, launches a headless agent there, and tells you to walk away. Come back any time and type /bring-back.

Want to try it first without setting up a real dev box?

There’s a Docker-based fake remote you can spin up locally. It mirrors your username and $HOME, trusts your SSH key, and ships a stub claude binary so the round trip works end-to-end.

./scripts/dev-remote.sh up # spin up the test remote + rewrite config
./scripts/dev-remote.sh down # clean up when you're done

Heads up

Local edits during a send-away are discarded

Once you’ve sent the project away, the remote is the live copy. Anything you type or edit locally in the meantime — including Claude’s own trailing turns — gets overwritten when you bring the project back. The CLI prompts y/N first.

Linux + FUSE only, for now

Outpost mounts a FUSE overlay on ~/.claude/, so it needs a Linux kernel with unprivileged FUSE (Ubuntu 22.04+, Debian 12+, Fedora 36+, Arch). macOS could work via macFUSE later. Windows is not on the roadmap.

One remote at a time

One outpost daemon talks to one configured dev box. No fan-out, no load balancing, no orchestration. If you want a different remote for a different project, you swap the config and reload.
